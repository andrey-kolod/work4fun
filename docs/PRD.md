## 1. Введение

### 1.1 Цель документа

```
Этот PRD определяет полные требования к системе управления задачами **Work4Fun версии 1.0 (MVP)**.
Документ служит единым источником истины для команды разработки, тестировщиков и стэкхолдеров.
```

### 1.2 Обзор продукта

```
Work4Fun — веб-приложение на Next.js для управления задачами в малых и средних командах (до 50
пользователей). Ключевые особенности:

- ✅ **Глобальные роли**: Супер-админ (SUPER_ADMIN), Обычный пользователь (USER)
- ✅ **Локальные роли в проекте**: Владелец проекта (PROJECT_OWNER), Администратор проекта
  (PROJECT_ADMIN), Участник проекта (PROJECT_MEMBER), Администратор группы (GROUP_ADMIN), участник
  группы (GROUP_MEMBER)
- ✅ **Канбан-доска**: Drag-and-drop задачи по статусам
- ✅ **Умные уведомления**: In-app, email, Telegram (админ управляет получателями)
- ✅ **Делегирование**: Передача задач между сотрудниками
- ✅ **Ограничения**: Максимум 3 проекта на пользователя с ролью PROJECT_OWNER
- ✅ **Группы внутри проекта**: Разделение задач по каналам (например, Магазин, Wildberries, Ozon)
- ✅ **Вся история**: Логирование всех действий, история статусов
```

### 2.2 Детализация прав и ограничений

#### 2.2.1 Глобальные роли

```
- SUPER_ADMIN — Полный доступ ко всей системе.
- USER — Базовая глобальная роль всех зарегистрированных пользователей.
```

#### 2.2.2 Локальные роли в проекте

```
Роли назначаются только внутри конкретного проекта (через таблицу ProjectMembership):

────────────────────────────────────────────
1. ВЛАДЕЛЕЦ ПРОЕКТА (PROJECT_OWNER)
────────────────────────────────────────────
Права в СВОИХ проектах:
• ✅ Создание проектов (макс. 3 активных)
• ✅ Приглашение пользователей в проект
• ✅ Назначение ролей PROJECT_ADMIN, PROJECT_MEMBER, GROUP_ADMIN и GROUP_MEMBER
• ✅ Создание/управление группами
• ✅ Полное управление задачами, участниками, настройками проекта
• ✅ Просмотр аналитики проекта
• ✅ Просмотр истории проекта

Ограничения:
• ❌ Не видит чужие проекты
• ❌ Нет доступа к системным настройкам

────────────────────────────────────────────
2. АДМИНИСТРАТОР ПРОЕКТА (PROJECT_ADMIN)
────────────────────────────────────────────
Делегированные права в проекте:
• ✅ Приглашение пользователей
• ✅ Назначение задач и управление группами
• ✅ Редактирование задач и комментариев
• ❌ Не может удалять проект или менять владельца

────────────────────────────────────────────
3. УЧАСТНИК ПРОЕКТА (PROJECT_MEMBER)
────────────────────────────────────────────
Права в проекте:
• ✅ Просмотр и выполнение задач в своих группах
• ✅ Изменение статуса своих задач
• ✅ Делегирование своих задач
• ✅ Комментирование задач

Запрещено:
• ❌ Приглашение пользователей
• ❌ Создание/удаление групп
• ❌ Доступ к аналитике

────────────────────────────────────────────
4. СУПЕР-АДМИН (SUPER_ADMIN)
────────────────────────────────────────────
• ✅ Полный доступ ко всем проектам и пользователям
• ✅ Системные настройки (SMTP, интеграции)
• ✅ Просмотр всех логов
• ✅ Управление всеми сущностями системы
```

#### 2.2.3 Особые случаи

```
• Разные роли в разных проектах:
    Пользователь может быть PROJECT_OWNER в Проекте А и PROJECT_MEMBER в Проекте Б.
• Лимит 3 проекта:
    Считается только количество проектов, где пользователь имеет роль PROJECT_OWNER.
    Участие как PROJECT_MEMBER или PROJECT_ADMIN не влияет на лимит.
• Группы внутри проекта:
    Задачи могут быть:
        Привязаны к конкретной группе (видят только участники этой группы)
        Общие (без группы) — видят все участники проекта (или только админы — по реализации)
        Личные (видят только исполнители (может быть более 1го исполнителя))
• Начальные роли при регистрации:
    Все новые пользователи получают глобальную роль USER.
    Роль PROJECT_OWNER получает только тот, кто создал проект (автоматически).
    Роль PROJECT_ADMIN назначается владельцем проекта.
```

#### 2.2.4 Начальные роли при регистрации

```
Самостоятельная регистрация (/register):
    • Глобальная роль: USER
    • Не состоит ни в одном проекте
    • Может создать проект (станет PROJECT_OWNER в нём)

Регистрация по приглашению:
    • Глобальная роль: USER
    • Локальная роль в проекте: PROJECT_MEMBER или PROJECT_ADMIN (по приглашению)

Назначение Супер-админом:
    • Может вручную назначить любую локальную роль в любом проекте
```

### 3.1 🔐 Авторизация и регистрация

#### 3.1.1 Регистрация пользователя в системе через форму регистрации

```
Пользователь имеет возможность перейти на форму регистрации нажатием на кнопку "Регистрация" на главной странице системы (/) или по прямой ссылке (/register).

Форма регистрации содержит следующие поля:

    - "Электронная почта" (обязательно):
        - Уникальный в системе;
        - Валидный формат электронной почты;
        - Максимум 255 символов.

    - "Имя" (обязательно):
        - Принимает буквы латинского и кириллического алфавитов, пробелы, дефисы и апострофы;
        - Принимает от 2 до 50 символов;
        - Автоматическое обрезание пробелов в начале и конце строки;
        - Валидация при вводе:
            - Если введены невалидные символы, поле подсвечивается красным цветом и пользователь видит сообщение: "Введены невалидные символы";
            - Если поле пустое, содержит только пробелы или менее 2 символов, поле подсвечивается красным цветом и пользователь видит сообщение: "Должно быть от 2 до 50 символов".

    - "Фамилия" (обязательно):
        - Принимает буквы латинского и кириллического алфавитов, пробелы, дефисы и апострофы;
        - Принимает от 2 до 50 символов;
        - Автоматическое обрезание пробелов в начале и конце строки;
        - Валидация при вводе:
            - Если введены невалидные символы, поле подсвечивается красным цветом и пользователь видит сообщение: "Введены невалидные символы";
            - Если поле пустое, содержит только пробелы или менее 2 символов, поле подсвечивается красным цветом и пользователь видит сообщение: "Должно быть от 2 до 50 символов".

    - "Телефон" (не обязательно):
        - Текстовое поле;
        - Принимает только цифры, скобки, пробелы, плюс.

    - "Пароль" (обязательно):
        - Пароль должен содержать минимум 8 символов;
        - Пароль должен содержать хотя бы одну букву латинского алфавита в верхнем и нижнем регистре;
        - Пароль должен содержать хотя бы одну цифру;
        - Независимая кнопка "Показать/скрыть пароль". По умолчанию скрыт;
        - Валидация требований при вводе. Если пароль не соответствует требованиям, поле подсвечивается красным цветом и пользователь видит сообщение: "Пароль должен содержать цифры и буквы латинского алфавита в верхнем и нижнем регистре".

    - "Подтвердить пароль" (обязательно):
        - Поле обязательно для заполнения;
        - Пароли должны совпадать.
        - Валидация требований при вводе. Если пароли не совпадают, поле подсвечивается красным цветом и пользователь видит сообщение: "Пароли не совпадают";
        - Независимая кнопка "Показать/скрыть пароль". По умолчанию скрыт.

    - "Пользовательское соглашение" (обязательно):
        - Чекбокс. Не активен по умолчанию;
        - Текст: "Я согласен с пользовательским соглашением";
        - Ссылка на пользовательское соглашение (/terms).

    - "Политика конфиденциальности" (обязательно):
        - Чекбокс. Не активен по умолчанию;
        - Текст: "Я согласен с политикой конфиденциальности";
        - Ссылка на политику конфиденциальности (/privacy-policy).

    - "Капча" (обязательно):
        - Компонент Google reCAPTCHA v3 или hCaptcha;
        - Виджет должен быть успешно решен пользователем для активации кнопки отправки формы.

    - Кнопка "Зарегистрироваться":
        - Изначально неактивна.
        - Становится активной только после:
            1. Прохождения базовой клиентской валидации всех обязательных полей (формат, длина).
            2. Отметки обоих обязательных чекбоксов.
            3. Успешного решения капчи.

Пользователь нажимает кнопку "Зарегистрироваться":
    - Проверка капчи: Если капча не была решена или решена неверно, отправка блокируется.
    - Отправка данных на сервер (`/api/auth/register`).
    - Серверная обработка:
        1. Валидация капчи (интеграция с сервисом, например, Google reCAPTCHA).
        2. Rate limiting: Проверка IP-адреса (макс. 100 запросов в минуту).
        3. Валидация данных: Повторная проверка форматов.
        4. Нормализация телефона: Если номер введен, он приводится к формату E.164 (только `+` и цифры) для хранения.
        5. Проверка уникальности `email` в базе данных.

    При успешном завершении регистрации:
        - Создается пользователь в базе данных с глобальной ролью USER и 'isEmailVerified = false';
        - Пользователю направляется письмо с ссылкой подтверждения регистрации;
        - Пользователь видит сообщение "Письмо с подтверждением регистрации отправлено на вашу электронную почту";
        - Пользователь переходит по ссылке подтверждения регистрации и перенаправляется на страницу авторизации (/login).

    При неудачном завершении регистрации пользователь получает сообщение об ошибке:
        - если почта уже зарегистрирована:
            - "Пользователь с такой электронной почтой уже зарегистрирован. Попробуйте восстановить пароль.":
            - На форме отображается подсвеченная ссылка "Восстановить пароль" (/password/reset);
            - Поле "Почта" подсвечивается красным цветом;
            - Введенный текст сохраняется в поле "Почта".
        - если email невалиден:
            - "Невалидная электронная почта. Попробуйте ещё раз.":
            - Поле "Почта" подсвечивается красным цветом;
            - Защита от атак на сервере ✅ Упростить: Rate limiting на уровне API (100 запросов/минуту на IP);
            - Введенный текст сохраняется в поле "Почта".
        - если более 100 запросов/минуту с одного IP:
            - "Слишком много запросов. Попробуйте через минуту.";
            - Отображается счетчик обратного отсчета времени на 60 секунд;
            - Кнопка "Зарегистрироваться" до завершения отсчета времени не активна.
        - если ошибка на сервере:
            - "Произошла ошибка на сервере. Попробуйте ещё раз.";
            - Кнопка "Зарегистрироваться" активна;
            - Сообщение об ошибке отображается ниже формы.

* Письмо с подтверждением регистрации:
    - Логотип Work4Fun;
    - Текст: "Для завершения регистрации подтвердите вашу электронную почту, нажав на кнопку ниже.";
    - Кнопка "Подтвердить email" (ведет на `/auth/verify?token=...`);
    - Примечание: "Если вы не создавали учетную запись, проигнорируйте это письмо.";
    - Контакты поддержки.

При отсутствии подтверждения регистрации пользователь будет удален из базы данных через 24 часа.
```

#### 3.1.5 Авторизация пользователя в системе

```
Для авторизации используется форма входа на странице `/login`.

Форма содержит:
    - "Электронная почта" (обязательно)
    - "Пароль" (обязательно) с кнопкой показа/скрытия
    - Чекбокс "Запомнить меня" (неактивен по умолчанию):
        * Если отмечен — сессия на 30 дней
        * Если не отмечен — сессия на 24 часа
    - Ссылка "Забыли пароль?" (/password/reset)
    - Кнопка "Войти" (активна после заполнения полей)

Пользователь нажимает "Войти":
    - Отправка на /api/auth/login
    - Серверная обработка:
    - Rate limiting: 10 попыток/минуту с IP, 5 попыток на email
    - Проверка существования пользователя
    - Проверка isActive = true (если false — ошибка "Аккаунт деактивирован")
    - Проверка isEmailVerified = true (если false — ошибка)

При успешном входе:
    - Создаётся JWT-токен (в httpOnly cookie)

При неудачном входе:
    - Неверные данные: "Неверный email или пароль."
    - Аккаунт неактивен: "Аккаунт деактивирован"
    - Email не подтверждён: соответствующее сообщение
    - Превышен лимит: блокировка с таймером

Дополнительно:
 - Все попытки входа логируются
 - Супер-админ видит логи всех пользователей
```

#### 3.1.7 Восстановление пароля

```
Пользователь может восстановить пароль через ссылку "Забыли пароль?" на странице входа (/login).

Процесс восстановления:

1. Запрос восстановления:
   - Пользователь переходит на `/password/reset`
   - Вводит email в поле "Электронная почта" (обязательно):
     - Валидный формат электронной почты
     - Максимум 255 символов
     - Клиентская валидация: если поле пустое или невалидный формат - подсветка красным и сообщение "Введите корректный email"
   - Нажимает кнопку "Отправить ссылку восстановления"

2. Проверка на сервере:
   - Rate limiting: максимум 5 запросов в час с одного IP, максимум 3 запроса на один email
   - Проверка существования пользователя с таким email
   - Проверка статусов:
     - Если `isActive = false`: возвращается сообщение "Аккаунт деактивирован. Обратитесь к Системному Админу"
     - Если `isEmailVerified = false`: возвращается сообщение "Email не подтверждён. Сначала подтвердите email"
     - Если `hasPasswordSet = false` (приглашённый пользователь): возвращается сообщение "Пароль ещё не установлен. Используйте ссылку из приглашения"
   - Если все проверки пройдены:
     - Генерация уникального токена восстановления (срок: 1 час)
     - Сохранение токена в базе данных с флагом `isUsed = false`
     - Отправка письма восстановления

3. Отправка письма:
   - Письмо содержит:
     - Логотип Work4Fun
     - Текст: "Вы запросили восстановление пароля для аккаунта в системе Work4Fun"
     - Кнопка "Восстановить пароль" (ведёт на `/password/reset/confirm?token=...`)
     - Примечание: "Ссылка действительна 1 час. Если вы не запрашивали восстановление пароля, проигнорируйте это письмо"
     - Контакты поддержки

4. Установка нового пароля:
   - Пользователь переходит по ссылке из письма на `/password/reset/confirm?token=...`
   - Страница содержит:
     - Информацию: "Установка нового пароля для [email пользователя]"
     - Поле "Новый пароль" (обязательно):
       - Минимум 8 символов
       - Должен содержать хотя бы одну букву латинского алфавита в верхнем и нижнем регистре
       - Должен содержать хотя бы одну цифру
       - Независимая кнопка "Показать/скрыть пароль"
       - Клиентская валидация: если требования не выполнены - подсветка красным и сообщение "Пароль должен содержать цифры и буквы латинского алфавита в верхнем и нижнем регистре"
     - Поле "Подтверждение пароля" (обязательно):
       - Должно совпадать с полем "Новый пароль"
       - Независимая кнопка "Показать/скрыть пароль"
       - Клиентская валидация: если пароли не совпадают - подсветка красным и сообщение "Пароли не совпадают"
     - Кнопка "Установить новый пароль" (активна только при заполнении обоих полей и прохождении валидации)

   - При нажатии кнопки:
     - Проверка валидности токена: не просрочен (≤1 час), не использован ранее (`isUsed = false`)
     - Валидация пароля по требованиям (повторная серверная проверка)
     - Если всё успешно:
       - Обновление хэша пароля в базе данных
       - Установка `isUsed = true` для токена восстановления
       - Инвалидация всех активных сессий пользователя (выход со всех устройств)
       - Отправка email-уведомления: "Пароль для вашего аккаунта был изменён"
       - Перенаправление на страницу входа (`/login`) с сообщением "Пароль успешно изменён. Войдите с новым паролем"

   - Если токен невалиден:
     - Показывается сообщение: "Ссылка восстановления недействительна или просрочена. Запросите новую ссылку"
     - Кнопка "Запросить новую ссылку" (ведёт обратно на `/password/reset`)

Защитные меры:
    - Токен восстановления можно использовать только один раз
    - После успешной смены пароля все предыдущие сессии пользователя сбрасываются
    - При 3 неудачных попытках ввода нового пароля токен блокируется (устанавливается `isUsed = true`)
    - Все запросы на восстановление логируются с указанием:
        - Email пользователя
        - IP-адрес
        - Время запроса
        - Статус (успешно/неудача/причина)
    - Супер-Админ видит все логи восстановления паролей

Особые случаи:
    1. Пользователь не получил письмо:
        - На странице `/password/reset` после отправки: "Если вы не получили письмо, проверьте папку 'Спам' или запросите новую ссылку через 5 минут"
        - Кнопка "Отправить повторно" (доступна через 5 минут после предыдущего запроса)

    2. Множественные запросы:
        - Если пользователь запросил новую ссылку, старая ссылка становится недействительной
        - Действует только последний отправленный токен

    3. Попытка восстановления при активной сессии:
        - Если пользователь уже авторизован и пытается восстановить пароль - перенаправление на страницу смены пароля в профиле (`/profile/change-password`)
        - Сообщение: "Вы уже авторизованы. Для смены пароля перейдите в настройки профиля"
```

#### 3.1.8 Управление доступом пользователей

```
Система предоставляет разные уровни управления доступом пользователей в зависимости от роли:

────────────────────────────────────────────
1. УПРАВЛЕНИЕ ДЛЯ РУКОВОДИТЕЛЯ (в рамках своего проекта)
────────────────────────────────────────────

Руководитель может управлять пользователями ТОЛЬКО в своих проектах:

1. Деактивация в проекте:
   - В разделе "Участники проекта" → выбор пользователя → "Деактивировать в проекте"
   - Предупреждение: "Пользователь потеряет доступ к этому проекту, но сохранит доступ к системе и другим проектам"
   - После деактивации:
     * Пользователь удаляется из активных участников проекта
     * Устанавливается `isActiveInProject = false`
     * Сохраняется вся история пользователя в проекте (задачи, комментарии)
     * Пользователь получает уведомление: "Ваш доступ к проекту «{projectName}» отозван"

2. Активация в проекте (восстановление):
   - В разделе "Неактивные участники" проекта
   - Выбор пользователя → "Восстановить доступ"
   - Пользователь снова становится активным участником проекта
   - Отправляется уведомление: "Ваш доступ к проекту «{projectName}» восстановлен"
   - Все предыдущие задачи и активность сохраняются

3. Удаление из проекта:
   - В разделе "Участники проекта" → выбор пользователя → "Удалить из проекта"
   - Предупреждение с выбором:
     * "Сохранить историю" (рекомендуется):
       - Данные пользователя остаются в истории проекта
       - Отображается как "Удалённый пользователь"
       - Можно пригласить заново
     * "Удалить полностью":
       - Все данные пользователя удаляются из проекта
       - Задачи перепривязываются к "Удалённый пользователь"
       - Необратимая операция
   - Требуется подтверждение: ввод текста "УДАЛИТЬ"
   - После удаления:
     * Пользователь исключается из проекта
     * Может быть приглашён повторно

Ограничения Руководителя:
• Не может деактивировать/удалять других Руководителей проекта (только Системный Админ)
• Не может управлять пользователями вне своих проектов

────────────────────────────────────────────
2. УПРАВЛЕНИЕ ДЛЯ СИСТЕМНОГО АДМИНА (глобальное)
────────────────────────────────────────────

Системный Админ может управлять всеми пользователями:

1. Глобальная деактивация:
   - В админ-панели (/admin/users) → выбор пользователя → "Деактивировать"
   - Предупреждение: "Пользователь потеряет доступ ко всей системе и всем проектам"
   - После деактивации:
     * `isActive = false`
     * Все активные сессии инвалидируются
     * Пользователь удаляется из всех проектов
     * Все данные сохраняются

2. Глобальная активация (восстановление):
   - В разделе "Деактивированные пользователи"
   - Выбор пользователя → "Активировать"
   - Пользователь получает доступ ко всем проектам, в которых состоял
   - Требуется повторное подтверждение email

3. Удаление из системы (полное):
   - В админ-панели → выбор пользователя → "Удалить из системы"
   - Многоступенчатое подтверждение:
     1. Предупреждение: "Все данные пользователя будут удалены безвозвратно"
     2. Проверка: пользователь не должен быть активным в проектах
     3. Ввод текста "УДАЛИТЬ НАВСЕГДА"
     4. Подтверждение паролем Системного Админа
   - После удаления:
     * Все личные данные пользователя удаляются из базы
     * Задачи и комментарии анонимизируются ("Удалённый пользователь")
     * Логи активности сохраняются (для аудита)
     * Необратимая операция

4. Управление в проектах:
    - Системный Админ может выполнять все те же действия, что и Руководитель в любом проекте
    - Может деактивировать/удалять Руководителей в проектах
    - Может назначать новых Руководителей

────────────────────────────────────────────
3. УПРАВЛЕНИЕ ДЛЯ ПОЛЬЗОВАТЕЛЯ (самостоятельное)
────────────────────────────────────────────

1. Удаление своего аккаунта:
    - В настройках профиля (/profile/settings) → "Удалить аккаунт"
      - Процесс удаления:
        1. Многоступенчатое подтверждение:
           - Предупреждение: "Внимание! Это действие необратимо:"
           - "• Все ваши личные данные будут удалены"
           - "• Вы потеряете доступ ко всем проектам"
           - "• Ваши задачи и комментарии будут анонимизированы"
        2. Требуется ввод текущего пароля
        3. Ввод подтверждающего текста "УДАЛИТЬ МОЙ АККАУНТ"
        4. Чекбокс: "Я понимаю, что это действие необратимо"

    - После подтверждения:
        * Устанавливается период ожидания 24 часа: `isMarkedForDeletion = true`, `deletionDate = now + 24h`
        * Пользователь теряет доступ ко всем проектам
        * Отправляется email с подтверждением и ссылкой для отмены
        * В течение 24 часов пользователь может отменить удаление

    - Через 24 часа:
        * Аккаунт автоматически удаляется
        * Все личные данные удаляются
        * Задачи и комментарии помечаются как "Удалённый пользователь"
        * Отправляется финальное уведомление об удалении

2. Отмена удаления (в течение 24 часов):
        - По ссылке из письма-подтверждения
        - Или в настройках профиля (если пользователь ещё может войти)
        - Аккаунт полностью восстанавливается
          * `isMarkedForDeletion = false`
          * Восстанавливается доступ ко всем проектам

────────────────────────────────────────────

1. Защита от случайных действий:
   - Все операции удаления требуют многоступенчатого подтверждения
   - По умолчанию предлагается деактивация вместо удаления
   - Для необратимых операций требуется ввод подтверждающего текста

2. Логирование:
   - Все операции логируются с указанием: кто, кого, когда, тип операции

      * Руководителю: логи действий пользователей в его проектах
      * Пользователю: доступ к системным логам не предоставляется (только история своих действий через интерфейс)

3. Уведомления:
   - При деактивации/удалении пользователь получает email с информацией
   - При восстановлении/активации — уведомление о восстановлении доступа

4. Особые случаи:

   - Системный Админ не может удалить самого себя
   - Приглашённые пользователи (без пароля) удаляются автоматически через 7 дней
   - Данные удалённых пользователей хранятся в архиве 30 дней для возможного восстановления (только Системным Админом)
```

#### 3.2.1 Создание проекта

```
Пользователь может создать новый проект при условии, что не достигнут лимит в 3 проекта с ролью PROJECT_OWNER.

Процесс создания проекта:

1. Доступ к форме создания:
    - Кнопка "Создать проект" доступна на:
        * Дашборде проектов (/projects)
        * Странице "Нет проектов" (если пользователь не состоит ни в одном проекте)
    - Перед открытием формы система проверяет PermissionService.canCreateProject(userId)
    - Если лимит превышен:
        * Кнопка "Создать проект" становится неактивной
        * При нажатии показывается сообщение: "Вы достигли лимита в 3 проекта. Чтобы создать новый, передайте владение одним из существующих проектов или удалите проект"

2. Форма создания проекта:
    - "Название проекта" (обязательно):
        * Текстовое поле
        * Минимум 3 символа, максимум 100 символов
        * Разрешены буквы, цифры, пробелы, дефисы и точки
        * Клиентская валидация: если менее 3 символов - подсветка красным и сообщение "Минимум 3 символа"
        * Счётчик символов (500/500)
        * Проверка уникальности названия проекта только в рамках проектов текущего владельца
        * Добавлено поле "Slug проекта" (автоматически генерируется из названия, транслитерация, уникальность глобально) — для красивых URL (/projects/my-project-slug) и SEO

    - "Описание проекта" (не обязательно):
        * Текстовое поле
        * Максимум 500 символов

    - Кнопка "Создать проект":
        * Изначально неактивна
        * Становится активной после заполнения обязательного поля "Название проекта" с валидными данными

3. Обработка создания:
    - При нажатии кнопки "Создать проект":
        * Отправка POST-запроса на `/api/projects`
        * Серверная проверка:
            1. Повторная проверка PermissionService.canCreateProject(userId)
            2. Валидация данных (длина, формат)
            3. Проверка уникальности названия проекта в рамках владения пользователя (нельзя создать два проекта с одинаковым названием одному пользователю)
            4. Добавлена проверка уникальности slug глобально (нельзя два проекта с одинаковым slug)

    - При успешном создании:
        * В базе данных создаются:
            - Проект (Project) с ownerId = текущий пользователь
            - Запись в ProjectMembership с role = PROJECT_OWNER
        * Пользователь перенаправляется на страницу проекта (`/projects/{projectId}`)
        * Показывается сообщение: "Проект '{название}' успешно создан!"
        * Добавлен toast-уведомление (sonner или react-hot-toast) для мгновенного подтверждения успеха

    - При ошибке:
        * Если лимит превышен: "Достигнут лимит в 3 проекта"
        * Если название неуникальное: "Проект с таким названием уже существует"
        * Общая ошибка: "Не удалось создать проект. Попробуйте ещё раз"

4. Особые случаи:
    - Для SUPER_ADMIN:
        * Нет ограничения в 3 проекта
        * Может создать проект без привязки к себе как owner (для тестирования)
    - При создании проекта пользователь автоматически получает роль PROJECT_OWNER
    - Проект создаётся в статусе "Активный"
    - Добавлена запись в ActivityLog: "Пользователь {имя} создал проект '{название}'"

```

#### 3.2.4 Удаление проекта

```
Только PROJECT_OWNER может удалить свой проект. SUPER_ADMIN может удалить любой проект.

1. Инициация удаления:
    - В настройках проекта: кнопка "Удалить проект"
    - Предупреждение: "Это действие необратимо!"

2. Многоступенчатое подтверждение:
    Шаг 1: Предупреждение
        - "Вы собираетесь удалить проект '{название}'"
        - Последствия:
            * Все задачи будут удалены
            * Все группы будут удалены
            * Все участники потеряют доступ
            * Все данные будут безвозвратно удалены через 30 дней

    Шаг 2: Выбор действия с данными:
        - "Удалить полностью" (рекомендуется для тестовых проектов)
        - "Архивировать вместо удаления":
            * Проект помечается как архивный
            * Данные сохраняются на 90 дней
            * Участники теряют доступ
            * Можно восстановить в течение 90 дней

    Шаг 3: Подтверждение:
        - Ввод текста "УДАЛИТЬ ПРОЕКТ {название проекта}"
        - Для SUPER_ADMIN: дополнительно ввод пароля

    Шаг 4: Финальное предупреждение:
        - Показывается список активных участников
        - "Эти пользователи потеряют доступ к проекту"
        - Кнопка "Уведомить участников" (отправляет email)

3. Процесс удаления:
    - Промежуточный статус "Удаляется" (24 часа)
    - В течение 24 часов:
        * PROJECT_OWNER может отменить удаление
        * Участники видят проект как "Удаляемый"
        * Новые задачи создать нельзя

    - Через 24 часа:
        * Все данные удаляются из основной базы
        * Перемещаются в архивную базу на 30 дней
        * Отправляются уведомления участникам

    - Через 30 дней:
        * Полное удаление из архивной базы

4. Отмена удаления:
    - В течение 24 часов PROJECT_OWNER может отменить удаление
    - В настройках проекта: кнопка "Восстановить проект"
    - Все данные возвращаются
    - Участникам восстанавливается доступ

    Упрощение процесса (лучшая практика продакшена): Soft delete (поле deletedAt в Project).
// - Нет промежуточного "Удаляется" (сложно с cron).
// - Восстановление до 30 дней (кнопка "Восстановить" в списке удалённых).
// - Архивная копия — через deletedAt + отдельная страница "Удалённые проекты".
// - Безопасно: Данные не удаляются физически сразу.
// - Проще реализовать (нет cron job).
```

#### 3.2.5 Управление участниками проекта

```
PROJECT_OWNER и PROJECT_ADMIN могут управлять участниками проекта.

1. Список участников:
    - Страница: `/projects/{projectId}/members`
    - Таблица участников:
        * Фото/аватар
        * Имя и фамилия
        * Email
        * Телефон
        * Роль в проекте
        * Дата присоединения
        * Статус (Активный / Неактивный)
        * Действия (для админов)

    - Фильтры:
        * По роли: Все / Владелец / Админ / Участник
        * По статусу: Активные / Неактивные
        * Поиск по имени или email

2. Приглашение новых участников:
    - Кнопка "Пригласить участника"
    - Форма приглашения:
        * "Email участника" (обязательно)
        * "Роль в проекте": PROJECT_MEMBER / PROJECT_ADMIN
        * "Персональное сообщение" (не обязательно)

    - Процесс приглашения:
        1. Проверка: не приглашался ли уже этот email
        2. Если пользователь уже в системе:
            * Добавляется в проект с выбранной ролью
            * Получает уведомление в системе и по email
        3. Если пользователя нет в системе:
            * Создаётся запись пользователя с role=USER
            * Отправляется приглашение с установкой пароля
            * После установки пароля добавляется в проект

    - Ограничения:
        * Только PROJECT_OWNER может назначать роль PROJECT_ADMIN
        * Нельзя пригласить пользователя, который уже в проекте
        * Максимум 50 участников в проекте (настраивается администратором)
        * Реализация лимита через PermissionService.canAddMember(projectId, userId) — count(members) < 50

3. Изменение ролей:
    - В таблице участников: выпадающий список "Изменить роль"
    - PROJECT_OWNER может:
        * Назначать PROJECT_ADMIN
        * Назначать PROJECT_MEMBER
        * Снимать роль PROJECT_ADMIN
    - PROJECT_ADMIN может:
        * Назначать PROJECT_MEMBER
        * Не может управлять PROJECT_ADMIN и PROJECT_OWNER

    - При изменении роли:
        * Запись в ActivityLog
        * Уведомление пользователю
        * Автоматическая перепривязка прав

4. Удаление участников:
    - В таблице: кнопка "Удалить из проекта"
    - Подтверждение: "Удалить {имя} из проекта?"
    - Варианты удаления:
        * "Только из проекта" (участник остаётся в системе)
        * "Заблокировать доступ" (не может быть приглашён повторно)

    - Последствия удаления:
        * Все задачи пользователя переходят в статус "Без исполнителя"
        * Комментарии сохраняются
        * История активности и бывшего исполнителя сохраняется
        * Если пользователь без проектов не входит в систему больше 180 дней, то удаляется из системы.
        * Добавлено поле blockedInProject в ProjectMembership (boolean) для блокировки

5. Статистика участников:
    - Количество участников по ролям
    - Активность участников (последний вход, выполненные задачи)
    - График вовлечённости
```

#### 3.2.6 Передача владения проектом

```
PROJECT_OWNER может передать владение проектом другому участнику.

1. Условия передачи:
    - Текущий владелец должен иметь роль PROJECT_OWNER
    - Новый владелец должен уже быть участником проекта
    - Новый владелец не должен достигнуть лимита в 3 проекта

2. Процесс передачи:
    Шаг 1: Инициация
        - В настройках проекта: "Передать владение"
        - Выбор участника из списка (только активные участники)
        - Подтверждение: "Вы передаёте владение проектом {участнику}"

    Шаг 2: Подтверждение нового владельца
        - Новому владельцу отправляется запрос на подтверждение
        - Email и in-app уведомление
        - Срок ответа: 7 дней

    Шаг 3: Передача
        - После подтверждения нового владельца:
            * Старый владелец становится PROJECT_ADMIN
            * Новый владелец становится PROJECT_OWNER
            * Владелец меняется в записи проекта
            * Запись в ActivityLog

    Шаг 4: Уведомления
        - Всем участникам проекта отправляется уведомление
        - "При смене владельца проект остается у участников.
        - Если новый владелец не подтвердит принятие в течение 7 дней:
            - Владельцем становится самый старый активный участник проекта, который входил в состему последние 30 дней
            - Отправляется уведомление о предложении возглавить проект новому владельцу"

3. Отмена передачи:
    - В течение 7 дней текущий владелец может отменить передачу
    - После отмены процесс начинается заново

4. Особые случаи:
    - Если новый владелец отказывается: процесс отменяется
    - Если новый владелец не отвечает 7 дней: процесс отменяется
    - Если у нового владельца уже 3 проекта: ошибка "Невозможно передать владение"

5. Автоматическая передача:
    - Если PROJECT_OWNER деактивирован или удалён:
        * Система ищет PROJECT_ADMIN в проекте
        * Если есть - передаёт владение первому админу
        * Если нет - передаёт супер-админу
        * Запись в системные логи
        * Реализация через поля pendingOwnerId, pendingOwnerExpiresAt в Project + cron job (ежедневно)
```

```
4. Восстановление из архива:

    - Участникам возвращается доступ
    - Поле status в Project: 'ACTIVE' | 'ARCHIVED' (enum) — проще, чем отдельный флаг
```

### 4.2 🔒 Безопасность

```
🛡️ Многоуровневая система защиты от брут-форса и злоупотреблений:

1. Rate Limiting на основе Redis (sliding window):
   • Логин по IP: ${RATE_LIMIT_LOGIN_IP:-10/60} (по умолчанию 10 попыток в 60 секунд)
   • Логин по email: ${RATE_LIMIT_LOGIN_EMAIL:-5/60} (по умолчанию 5 попыток в 60 секунд)
   • Регистрация: 100 попыток в минуту с одного IP
   • Восстановление пароля: 5 запросов в час с IP, 3 запроса на email

2. Реализация:
   • Redis Sorted Sets для sliding window алгоритма
   • Автоматическая очистка устаревших записей
   • Fail-open стратегия: при недоступности Redis запросы пропускаются
   • Детальное логирование блокировок с указанием причины (ip/email)

3. Механизм блокировки:
   • Возвращает retryAfter (секунды до разблокировки)
   • Указывает blockedBy ('ip' или 'email') для аналитики
   • Логирует попытки с маскировкой персональных данных

4. Настройка через переменные окружения:
   RATE_LIMIT_ENABLED=true/false
   RATE_LIMIT_LOGIN_IP=10/60
   RATE_LIMIT_LOGIN_EMAIL=5/60

5. Мониторинг в Grafana:
   • Метрики rejected_connections из redis-exporter
   • Логи блокировок в Loki (уровень WARN)
   • Дашборд с количеством блокировок по типам
```

### 5.4 Логирование (Loki + Promtail)

```
📜 Единая система логирования на базе Loki:

Источники логов:
1. Docker контейнеры (автообнаружение через Docker Socket):
   • Все сервисы мониторинга

   • Метки: job="work4fun-app", source="file", level, service_name

3. Системные логи:
   • Ротация: 24 часа индекса, хранение в volumes
   • Формат: JSON с полным стектрейсом ошибок

Конвейер обработки (Promtail):
# Для файлов приложения
pipeline_stages:
  - json:
      expressions:
        log_level: level
        service: service
        message: message
        timestamp: timestamp
  - timestamp:
      source: timestamp
      format: RFC3339
  - labels:
      level: log_level
      service_name: service

Запросы в Loki (Grafana Explore):
# Все логи приложения

# Логи rate limiting
{service_name="work4fun"} |= "blockedBy"

# Поиск по пользователю

Производительность:
• Индексирование по меткам (уровень, сервис, контейнер)
```

```
Структура дашборда "🎛️ Monitoring Dashboard" (UID: prod-monitoring-dashboard):

   • Connected Clients — количество подключений

   • Rejected Connections (5 min) — отклонённые подключения
   • Commands/sec — нагрузка на Redis
   • Cache Hit Rate % — эффективность кэша

Устранение неполадок:

3. Логи не отображаются:
   • Проверьте файлы логов: ls -la monitoring/logs/

   • Состояние Loki: docker compose logs loki

   docker exec -it ${SERVICE_NAME}-grafana grafana-cli admin reset-admin-password новый_пароль
```
